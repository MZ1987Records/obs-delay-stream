# Technical Notes

## OBSコールバックサイズとOpusフレームサイズの不一致によるバッファ揺れ

OBSの音声コールバックは1024サンプル単位（48kHz時 ≈ 21.3ms）で呼ばれるが、
Opusエンコーダのフレームサイズは960サンプル（48kHz時 = 20ms）である。

この不一致により、FIFO経由でエンコードする際に約15回に1回、
1回のコールバックから2つのOpusフレームが同時に生成・送信される。

```
コールバック1:  FIFO 0→1024,   エンコード→1パケット (残64)
コールバック2:  FIFO 64→1088,  エンコード→1パケット (残128)
...
コールバック15: FIFO 896→1920, エンコード→2パケット (残0)
```

レシーバー側では、2パケットがほぼ同時に到着するとバッファ表示が一時的に跳ね上がり、
その後徐々にドレインして設定値に戻る。

- 再生バッファを20msに設定した場合、接続直後は平均30ms付近に見えることがある
- しばらく再生を続けると20msに安定する
- 最小バッファは設定値どおり保たれており、実害はない

関連コード: `stream-router.cpp` `encode_opus_packets()` / `receiver/src/audio.ts` `playBuffer()`

## 送受信間のサンプルレート不一致（48000Hz → 44100Hz 環境など）

送信側のOpusエンコーダは `48000, 24000, 16000, 12000, 8000` Hz のいずれかで動作する
（`is_valid_opus_sample_rate()` 参照）。44100Hz は有効なOpusレートに含まれない。

一方、受信側ブラウザの `AudioContext` はOSのオーディオデバイスに合わせて
44100Hz で動作する場合がある（Windows/macOS のデフォルト設定による）。

この不一致があっても **遅延蓄積・音の歯抜けは発生しない**。理由:

1. `playBuffer()` のスケジューリングは秒単位で行われている
   ```typescript
   state.nextTime += abuf.duration;  // duration = frames / bufferSampleRate（秒）
   ```
   ハードウェアのサンプルレートに依存しない。

2. Web Audio API が `AudioBufferSourceNode` 再生時に
   バッファのサンプルレート → AudioContext のサンプルレートへ自動リサンプリングする。
   （例: 48000Hz AudioBuffer → 44100Hz AudioContext で透過的に変換）

3. パケットヘッダにサンプルレートが埋め込まれており（`[4B SR]`）、
   受信側はパケットごとに正しいレートで `AudioBuffer` を生成している。

結論: サンプルレートの不一致は Web Audio API の仕様内で吸収されるため、
追加のリサンプリング処理は不要。

関連コード: `receiver/src/audio.ts` `playBuffer()` `handlePcm16()` / `receiver/src/opus.ts` デコーダ初期化

## 再同期時ノイズ低減は「非重複ランプ + しきい値判定」が現実的

再同期時のクリックノイズは、旧ストリーム終端と新ストリーム先頭の波形不連続で発生する。
ただし本プロジェクトの再同期では、旧ストリームを切ってから新ストリームが始まるため、
**追加遅延ゼロを優先するならクロスフェードは採れない**。

実装上は、受信側の再生キューに対して次の方針が現実的だった。

1. 再同期時に旧キューを短時間ランプアウトして停止する
2. 次の先頭バッファだけ短時間ランプインする
3. 通常再生中の各パケットには処理を入れない

重要なのは、ランプ適用判定に `nextTime - currentTime` をそのまま使わないこと。
この値は通常時でも `AHEAD` 付近を保つため、しきい値比較に使うと毎回ランプしてしまう。

正しい判定は、**基準バッファ `AHEAD` からどれだけズレたか**で見ること。

```typescript
const bufferedSec = Math.max(0, state.nextTime - now);
const driftSec = Math.abs(bufferedSec - AHEAD);
if (driftSec < threshold) {
  // ランプしない
}
```

今回の調整では、以下が妥当だった。

- ランプイン長: `8ms`
- ランプアウト上限: `120ms`
- ランプ適用最小ズレ: `20ms`（`AHEAD` 基準差分）

また、自動再同期が有効なときは、手動再同期でも**カウント値だけでなくタイマー位相自体を再起動**する必要がある。
`syncCountdown` の値だけ戻しても、次の tick 時刻が据え置きだと表示上ほぼ変化しないことがある。

関連コード: `receiver/src/ui.ts` `resync()` `startAutoSync()` / `receiver/src/audio.ts` `playBuffer()`
